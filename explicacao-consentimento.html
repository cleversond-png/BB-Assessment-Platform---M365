Por que o cliente precisa conceder consentimento?

A plataforma lê configurações de governança do ambiente Microsoft 365 do cliente via Microsoft Graph API — o mesmo canal oficial usado pelas ferramentas da própria Microsoft. Nenhum email, arquivo, mensagem ou dado de conteúdo é acessado ou armazenado.

Princípio fundamental: Least Privilege absoluto
Cada permissão foi escolhida para cobrir exatamente o que precisa ser avaliado — nada mais. A App Registration não tem permissões de escrita em nenhum serviço. Não pode criar usuários, não pode enviar emails, não pode modificar políticas.
ℹ️
Como funciona o consentimento de administrador
O Global Admin do tenant do cliente acessa uma URL gerada pela plataforma, faz login com a conta corporativa e aprova as permissões listadas. A partir daí, a plataforma obtém tokens de acesso somente para leitura para aquele tenant específico. Os dados de outros clientes ficam completamente isolados.
⚠️
Duas permissões novas em V2 (re-consent necessário)
Se o cliente já concedeu consentimento na V1, será necessário re-consent para ativar: Application.Read.All (detecta apps de terceiros com permissões excessivas) e Team.ReadBasic.All (avalia configurações de segurança do Microsoft Teams). Ambas são somente leitura e estão detalhadas na aba "Permissões em Detalhe".

Permissões em Detalhe
Explicação técnica e em linguagem de negócio de cada permissão solicitada.
📋
AuditLog.Read.All
Governança
O que é
Permite ler o log de auditoria unificado do Microsoft 365 — o registro de todos os eventos administrativos e de login. É o equivalente a ler o livro de registros de entrada de um prédio: você sabe quando e quem entrou, mas não o que as pessoas falaram dentro.
Para que usamos
Detectar uso de protocolos de autenticação legados (SMTP, IMAP, POP3, Basic Auth) nos últimos 30 dias. Esses protocolos ignoram MFA e Conditional Access, representando risco crítico de comprometimento de conta.
O que lemos
  • Tipo de cliente de autenticação usado
  • UPN do usuário (para identificar contas expostas)
  • Data/hora do evento
O que não acessamos
  • Conteúdo de emails lidos
  • Senhas ou credenciais
  • Dados de negócio manipulados
🏢
Organization.Read.All & Reports.Read.All
Baseline
O que é
Organization.Read.All: lê o perfil da organização — nome do tenant, licenças Microsoft 365 ativas, configurações globais.

Reports.Read.All: lê relatórios de uso agregados dos serviços Microsoft 365 (quantos usuários ativos no Teams, SharePoint, etc.). Importante: a Microsoft anonimiza esses relatórios por padrão — não é possível saber quem especificamente usou o quê.
Para que usamos
Identificar quais licenças o cliente tem (para saber se Entra ID P1/P2 estão ativas), mapear adoção dos serviços, e detectar canais de comunicação em uso para avaliar risco de vazamento.
O que lemos
  • Nome e ID do tenant
  • Lista de SKUs de licença
  • Contagem de usuários ativos por serviço
O que não acessamos
  • Identidade de usuários específicos em relatórios
  • Conteúdo de qualquer comunicação
  • Dados financeiros ou de negócio
👥
User.Read.All
Baseline
O que é
Lê o diretório de usuários do Azure Active Directory / Entra ID. É o equivalente a acessar a lista telefônica interna da empresa: nome, email corporativo, departamento, status da conta.
Para que usamos
Identificar: contas de usuários guest (externos), usuários inativos há mais de 90 dias, proprietários de sites e grupos, e contas sem MFA configurado.
O que lemos
  • UPN, displayName, departamento
  • Data do último login
  • Tipo de conta (membro / guest)
  • Flag onPremisesSyncEnabled
O que não acessamos
  • Emails, calendários, arquivos de usuários
  • Senhas ou métodos de autenticação (outra permissão separada)
  • Dados de RH ou salários
🔐
UserAuthenticationMethod.Read.All
Identidade
O que é
Lê quais métodos de autenticação cada usuário tem registrados — ex.: "tem autenticador configurado?", "tem número de telefone para SMS?". Não lê senhas, tokens, ou segredos. É como saber que alguém tem um crachá, sem poder usá-lo.
Para que usamos
Calcular a cobertura de MFA (quantos % dos usuários têm fator adicional configurado) e a cobertura de SSPR — Self-Service Password Reset (quantos usuários podem redefinir a própria senha sem chamar o helpdesk).
⚡ Requer Entra ID P1 ou superior
O que lemos
  • Booleanos: isMfaRegistered, isSsprRegistered
  • Tipo de método (authenticator app, telefone, etc.)
O que não acessamos
  • Senhas em texto claro ou hash
  • Tokens de sessão ativos
  • Números de telefone completos
🛡️
Policy.Read.All
Identidade
O que é
Lê políticas de Conditional Access — as regras que definem quem pode acessar o quê, de onde, e com quais condições. É como ler o regulamento de segurança de um prédio: você vê as regras, mas não pode modificá-las.
Para que usamos
Avaliar se existem políticas que bloqueiam acesso de localizações suspeitas, que exigem dispositivo gerenciado, e para detectar contas break-glass (emergência) corretamente excluídas das políticas — mas ainda monitoradas.
⚡ Requer Entra ID P1 ou superior
👑
RoleManagement.Read.Directory
Identidade
O que é
Lê quais usuários estão atribuídos a funções administrativas do Entra ID — Global Admin, Exchange Admin, SharePoint Admin, etc. Não pode criar, remover ou alterar atribuições de função.
Para que usamos
Identificar excesso de administradores globais (risco crítico), verificar se contas privilegiadas têm MFA, e detectar contas break-glass que são Global Admins cloud-only (excluídas do Conditional Access para emergências).
⚠️
IdentityRiskyUser.Read.All
Identidade
O que é
Lê a lista de usuários que o próprio Microsoft Entra ID Protection marcou como "em risco" — baseado em machine learning que detecta comportamentos anômalos de login, credenciais vazadas, etc. São dados que a Microsoft já calculou; nós apenas lemos o resultado.
Para que usamos
Quantificar usuários com risco alto ou médio no tenant, o que indica possíveis contas comprometidas que precisam de ação imediata (reset de credenciais, investigação de sessões suspeitas).
⚡ Requer Entra ID P2
🔍
Application.Read.All
Identidade — Nova V2
O que é — explicação crítica para o cliente
Esta é frequentemente a permissão que gera mais dúvida, porque o nome pode ser confundido com "a app ganha acesso a tudo que outras apps têm acesso". Isso é falso.

O que Application.Read.All faz: Permite listar as aplicações registradas no Entra ID do cliente e ler quais permissões essas aplicações declararam precisar. É como auditar a lista de chaves que outras pessoas têm — você não ganha as chaves; você vê quem as tem.
Analogia: Imagine que sua empresa tem um cofre (o M365). Vários sistemas de terceiros (CRM, ERP, app de RH) pediram cópias de chaves desse cofre. Application.Read.All nos deixa ver a lista de quem tem cópia de chave e qual porta cada chave abre. Não nos dá nenhuma cópia de chave. Nos permite dizer ao cliente: "o sistema X tem acesso à sua caixa de emails, isso é aceitável?"
Para que usamos
Detectar aplicações de terceiros que receberam permissões de alto risco como: Mail.ReadWrite, Files.ReadWrite.All, Directory.ReadWrite.All. Essas apps, se comprometidas, representam risco crítico ao tenant — e muitas vezes os administradores nem sabem que elas existem.

Também detectamos secrets (senhas de apps) próximos de expirar — que quando expiram causam falhas silenciosas em integrações críticas.
O que lemos
  • Nome e ID das aplicações registradas
  • Lista de permissões que ELAS têm
  • Data de expiração de secrets de apps
O que não acessamos
  • Conteúdo acessado por essas apps
  • Tokens ou credenciais das apps
  • Capacidade de agir como essas apps
📂
Sites.Read.All · Files.Read.All · SharePointTenantSettings.Read.All
SharePoint
O que é — distinção crítica
Sites.Read.All: acessa metadados de sites (nome, proprietário, configurações de permissão, usuários com acesso). Files.Read.All: acessa metadados de arquivos (nome, proprietário, quando foi modificado, por quem foi compartilhado).

Nenhuma delas lê o conteúdo dos arquivos. A distinção na API do Microsoft Graph é clara: para ler conteúdo de arquivo, a permissão necessária seria diferente e usaria endpoints /content. Nosso código nunca chama esses endpoints.
Analogia: É como ter acesso ao sistema de controle de uma biblioteca. Você pode ver quem tem permissão de acessar cada sala e quais livros existem nas prateleiras, mas não pode abrir e ler nenhum livro.
Para que usamos
Identificar sites sem proprietário definido, detectar compartilhamentos externos excessivos (oversharing), verificar se o OneDrive está configurado para compartilhamento aberto externamente, e calcular o risco de vazamento de dados via links compartilhados.
🏷️
InformationProtectionPolicy.Read.All · RecordsManagement.Read.All
Governança
O que é
InformationProtectionPolicy.Read.All: lê as configurações dos rótulos de sensibilidade (Sensitivity Labels) do Microsoft Purview — ex.: "Confidencial", "Uso Interno", "Público". Não lê documentos rotulados, apenas a configuração dos rótulos.

RecordsManagement.Read.All: lê as políticas de retenção de dados configuradas — por exemplo, "emails de colaboradores devem ser mantidos por 7 anos". Não acessa os dados retidos.
Para que usamos
Verificar se o cliente tem classificação de dados configurada (requisito LGPD e ISO 27001) e se tem políticas de retenção que atendam requisitos regulatórios e de compliance.
🤖
ExternalConnection.Read.All
Copilot
O que é
Lê a lista de conectores externos configurados para o Microsoft Copilot — fontes de dados externas (como sistemas de CRM, bases de conhecimento, portais externos) que foram integradas ao Copilot via Microsoft Graph Connectors.
Para que usamos
Avaliar a extensibilidade do Copilot: o cliente já conectou fontes externas de conhecimento? Esses conectores têm governança adequada? Essa análise alimenta o score de "Copilot Readiness".
💬
Team.ReadBasic.All
Teams — Nova V2
O que é
Apesar do nome genérico, esta permissão dá acesso somente a configurações administrativas do Microsoft Teams — não a mensagens, canais, arquivos ou reuniões. Lê configurações como: acesso externo entre federações, permissões de instalação de apps de terceiros, e políticas de governança de apps.
Para que usamos
Verificar se o tenant permite federação irrestrita com qualquer organização externa (risco de exfiltração via chat), se usuários podem instalar apps de terceiros sem aprovação de admin, e se existe governança de aplicações no Teams.
O que lemos
  • Configuração de acesso externo (B2B)
  • Políticas de governança de apps
  • Configurações de segurança globais
O que não acessamos
  • Mensagens de chat ou canais
  • Arquivos compartilhados no Teams
  • Gravações de reuniões
  • Lista de membros de equipes específicas
Scripts por Perfil de Stakeholder
Respostas prontas para as objeções mais comuns, segmentadas por perfil de interlocutor.
🏛️
CTO / CISO
Estratégico
Por que vocês precisam de tantas permissões? Parece demais.
Resposta
Temos 15 permissões porque avaliamos 7 domínios de segurança distintos — identidade, compartilhamento, governança, email, Teams e prontidão para IA. Cada permissão cobre exatamente um domínio e foi escolhida por ser a menor permissão possível que entrega aquela análise. Nenhuma é Write. Para comparar: uma aplicação de produtividade típica como o Salesforce ou um CRM pede permissões Write em calendários e emails — nossa plataforma não escreve nada.
Se vocês têm Application.Read.All, isso significa que vocês podem fazer o que essas apps fazem?
Resposta
Não. Application.Read.All nos deixa ver a lista de apps e quais permissões elas têm — é uma permissão de auditoria, não de delegação. É como ler o inventário de chaves do seu departamento de facilities sem ganhar nenhuma das chaves listadas. Na API do Graph, para "agir como" outra app, seria necessário uma permissão completamente diferente, que não temos e não pedimos.
Como garantimos que os dados do nosso tenant não ficam misturados com os de outros clientes?
Resposta
A plataforma usa multi-tenant isolation por design: cada tenant tem seu próprio token de acesso, armazenado de forma isolada. O consentimento de cada cliente é específico para aquele tenant — o token gerado só serve para acessar aquele ambiente. Não há token compartilhado entre clientes. Os dados são armazenados por tenant ID com separação lógica completa.
O que acontece se eu revogar o consentimento depois?
Resposta
A revogação é imediata via portal Azure ou portal Microsoft 365. Ao revogar, o token expira na próxima chamada à API e a plataforma para de ter acesso ao ambiente. Os dados já coletados ficam disponíveis para consulta local, mas nenhum dado novo pode ser coletado. O processo é simples: Entra ID > Enterprise Applications > nossa app > Delete.

Jurídico / Compliance / DPO
Regulatório
Que dados pessoais vocês coletam? Isso viola a LGPD?
Resposta
Coletamos dados de configuração e metadados técnicos — UPNs (emails corporativos), nomes de usuários, status de MFA. Esses dados são usados exclusivamente para produzir o relatório de assessment e são tratados como dados corporativos, não dados sensíveis de pessoa física. A base legal é o legítimo interesse do controlador (art. 7º, IX, LGPD) para fins de diagnóstico de segurança contratado pela própria organização. Não coletamos conteúdo de comunicações, dados de saúde, dados financeiros pessoais ou qualquer dado sensível listado no art. 5º, II, LGPD.
Os dados ficam fora do Brasil? Isso viola a transferência internacional?
Resposta
Os dados são coletados via Microsoft Graph (infraestrutura Microsoft) e processados na plataforma da BB. A Microsoft já é coberta por cláusulas contratuais padrão e adequação para transferência internacional. A LGPD permite transferência internacional quando há consentimento do titular ou cláusulas específicas — no caso de dados corporativos de diagnóstico de segurança, o enquadramento é de serviço B2B contratado. Se necessário, podemos assinar um DPA (Data Processing Agreement) específico.
Como garantem que os dados do nosso assessment não vazam para concorrentes?
Resposta
Cada assessment é identificado por tenant ID e isolado logicamente. Os dados são armazenados com controle de acesso por tenant. Não há repositório compartilhado de dados entre clientes. O relatório gerado pertence ao cliente — a BB não usa dados de clientes para benchmarking ou treinamento de modelos sem consentimento explícito.

TI / Infraestrutura / Sysadmin
Técnico
Como sei que a app só está lendo e não escrevendo nada?
Resposta
Você pode verificar diretamente no portal Azure. Acesse: Entra ID → Enterprise Applications → [nome da app] → Permissions. Todas as permissões listadas terminam em .Read ou .Read.All — nenhuma termina em .Write, .ReadWrite ou .Manage. Você também pode ver o manifest JSON da App Registration e confirmar que não há nenhuma permissão de escrita declarada.
A app vai impactar performance ou disponibilidade do nosso M365?
Resposta
Não. A Microsoft Graph API é projetada para suportar cargas de consulta muito maiores que as nossas. A plataforma respeita os rate limits da API (throttling automático) e o assessment completo consome menos chamadas que uma sessão normal de usuário. O assessment dura aproximadamente 2-5 minutos e não afeta a experiência de usuários finais.
Preciso criar alguma conta de serviço ou abrir portas no firewall?
Resposta
Não é necessário criar conta de serviço, senha compartilhada, ou configurar firewall. O acesso é feito via OAuth 2.0 Client Credentials — fluxo app-to-app da Microsoft, sem envolvimento de usuário. O único requisito é que um Global Admin realize o consentimento uma única vez pela URL que geramos. Após isso, a plataforma funciona de forma autônoma com os tokens gerados.
Como revogo o acesso depois?
Resposta
Entra ID → Enterprise Applications → [nome da app] → Properties → Delete. A revogação é imediata. Alternativamente: App Registrations → [app] → API permissions → Revoke admin consent. Nenhuma ação adicional é necessária do lado da plataforma — o token para de funcionar no próximo uso.

Diretoria / Board
Executivo
O que estamos autorizando exatamente e qual o risco para a empresa?
Resposta
Estamos autorizando uma plataforma de diagnóstico a ler configurações técnicas do nosso ambiente Microsoft 365 — como um médico lendo os exames do paciente, não realizando cirurgia. A plataforma não pode enviar emails, deletar arquivos, criar usuários ou modificar qualquer configuração. O risco é equivalente ao de contratar uma auditoria externa: o valor entregue (identificação de riscos reais) supera amplamente o risco residual, e o acesso pode ser revogado a qualquer momento pelo nosso time de TI.
Por que precisamos fazer isso agora?
Resposta
Regulações como LGPD exigem medidas técnicas de segurança documentadas — e auditores perguntam especificamente sobre MFA, Conditional Access, classificação de dados e retenção. Este assessment produz evidência documentada do estado atual, identifica lacunas antes de um incidente ou auditoria, e cria o roadmap de remediação priorizado. O custo de não fazer é uma multa de até 2% do faturamento ou, pior, um incidente de segurança evitável.

O que Mail.ReadWrite e Files.ReadWrite.All significam no contexto do assessment

💡
Pergunta frequente: "Se vocês têm Application.Read.All e uma app no tenant tem Mail.ReadWrite — vocês conseguem ler emails?"
Não. Application.Read.All nos deixa saber que essa app tem Mail.ReadWrite. Não nos transfere a permissão. Para ter Mail.ReadWrite, precisaríamos ter solicitado essa permissão explicitamente — e ela não está na nossa lista de consentimento. A API do Microsoft Graph não permite delegação implícita de permissões.

Argumento final de segurança
Nossa plataforma detecta quando apps de terceiros têm essas permissões perigosas no tenant do cliente — e reporta isso como risco no assessment. Isso significa que estamos do mesmo lado que o time de segurança do cliente, não do lado das ameaças. Se a plataforma tivesse más intenções, não reportaria esse risco — seria uma vulnerabilidade que exploraria, não revelaria.
